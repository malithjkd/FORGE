## The Failure Spectrum

```
EXPERIMENT FAILURE                    PROCESS FAILURE
(the hypothesis was wrong)           (the system didn't support the work)
         │                                    │
         ▼                                    ▼
   Dead-End Entry (DE)              System Improvement Action
   → Feeds Knowledge Commons        → Feeds FORGE system health
   → Normal and expected            → Needs process change
```

## Key Principles

1. **A dead end is not a failure** — it is a confirmed negative result with the same value as a positive one
2. **Failure documentation is mandatory** — no exceptions (see SOP-004)
3. **Blame kills learning** — if people fear blame, they hide failures, and the system loses its most valuable input
4. **Conditions change** — a dead end today may become viable tomorrow with more data, better tools, or new techniques

## Key Questions to Address

- How do we distinguish "this approach fundamentally can't work" from "we didn't do it well enough"?
- What is the threshold for abandoning a track vs. re-scoping it?
- How do we prevent "dead-end fatigue" — where too many failures demoralise the team?
- How do we celebrate valuable dead ends (they saved future teams months of wasted effort)?
